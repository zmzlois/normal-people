**6\. Live Reload. Streaming. Revalidating. Time Interval. _All come together in_ [**_Next.js 13_**](https://beta.nextjs.org/docs/getting-started)**

 Benefits of [static-site generation](https://nextjs.org/docs/basic-features/pages#static-generation)(HTML has generated at **build time/ahead of time, reused on each request** and served from CDN geographically closer to the user), [server-side rendering](https://nextjs.org/docs/basic-features/pages#server-side-rendering)(also referred to as SSR or Dynamic rendering, HTML is generated on each request, can be configured in [segment config](https://beta.nextjs.org/docs/api-reference/segment-config)) and [incremental static regeneration](https://nextjs.org/docs/basic-features/data-fetching/incremental-static-regeneration)(create or update the static page without rebuilding the whole site) is now available through one API, and we can set the revalidating period(see example below).

```
// This request should be cached until manually invalidated.
// Similar to \`getStaticProps\`.
// \`force-cache\` is the default and can be omitted.
fetch(URL, { cache: 'force-cache' });

// This request should be refetched on every request.
// Similar to \`getServerSideProps\`.
fetch(URL, { cache: 'no-store' });

// This request should be cached with a lifetime of 10 seconds.
// Similar to \`getStaticProps\` with the \`revalidate\` option.
fetch(URL, { next: { revalidate: 10 } });

//in app/page.tsx
export const revalidate = 60; // revalidate this page every 60 seconds
```

**8\. The future is at the edge.**

# Why TurboPack

but not [Babel](https://babeljs.io/), [Terser](https://terser.org/), [esbuild](https://esbuild.github.io/), [swc](https://swc.rs/) and [Webpack](https://webpack.js.org/)?

#### **It is important enough to have a separate section to introduce it.**

> Using [TurboPack Alpha](https://turbo.build/pack) with Next.js 13 results in:
>
> **\- 700x faster** updates than Webpack
>
> **\- 10x faster** updates than Vite
>
> **\- 4x faster** cold starts than Webpack

To visualise/quantify this result: if you have 5,000 react components(which a lot of applications do), the cold start speed looks like this:

![Performance Comparison](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*v0H1Wnqaz8SWDNps2Tt5Vw.png)
Turbopack+Next.js 13 performance compare to Next.js 12, Vite and Next.js 11

There are two ways to make a process faster:  
1\. do less work;  
2\. do work in parallel.

Fast build system like [**Vite**](https://vitejs.dev/) **doesn‚Äôt bundle application source code in development mode** but rely on the browser‚Äôs native ES Modules system and resulting in incredibly responsive updates, but the bottleneck is scaling ‚Äî large applications are made up of many modules, and huge web traffic can results in a relatively slow start-up time. Vite might sound amazing 2‚Äì3 years ago, and Tailwind guys still love them; in the field of intelligent-build systems, it might not be the case anymore(who knows what will happen a couple of years down the line!).

Another thing about Vite is they utilise a supremely fast tool under the hood ‚Äî [esbuild](https://esbuild.github.io/). It only does one job ‚Äî bundling. Here is the catch: it doesn‚Äôt support [hot module replacement](https://webpack.js.org/concepts/hot-module-replacement/)(save [app state](https://reactjs.org/docs/state-and-lifecycle.html), only update what‚Äôs changed, native-speed level style change) in the development server; it doesn‚Äôt do much caching either, resulting in repetitive build ‚Äî slow down even at the speed of native. When esbuild is bundling, it‚Äôs all or nothing(unless you heavily configure it).

#### And how Turbopack redoes the job:

**1\. Lazy bundling**
Turbopack‚Äôs approach was to bundle the code in the development server, which can be achieved way faster than [Webpack](https://webpack.js.org/), especially for large applications. Webpack was written in Javascript, while **Turbopack was written in** [**Rust**](https://www.rust-lang.org/), a language created with **optimal performance, thread safety and incorruptible memory** in mind without runtime or garbage collection. Regarding performance stats, Rust is faster than Javascript on every benchmark(see below).

![Rust-vs-Javascript Performance
](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*4ScSnZW9IOdJQGCzi_azsA.png
)
[Rust-vs-Javascript Performance](https://programming-language-benchmarks.vercel.app/rust-vs-javascript)

Go back to lazy bundling, take an example when you request a home page, Turbopack will only bundle `app/page.tsx` and the modules related to it ‚Äî as minimal work as possible.

**2\. While bundling, it skips optimisation work which is only necessary for the production**

**3\. Incremental Computation**
Turbopack only computes the requested assets and caches the result of all the functions it did, so it never does the same job twice. You can read about it [here](https://turbo.build/pack/docs/core-concepts).

Performance benchmarks of [Turbopack vs Vite](https://turbo.build/pack/docs/comparisons/turbopack-vs-vite) and [Turbopack vs Webpack](https://turbo.build/pack/docs/comparisons/turbopack-vs-webpack) can be found by clicking on the links.

### Why TurboRepo(First of all you need to have monorepo in mind)

Hundreds-of-Apps. Single Codebase. High-Performance Build System.

**How many applications and modules you can stuff into one monorepo?** The answer is hundreds ‚Äî in extreme cases ‚Äî thousands. But to do it faster? There are multiple tools: Google has [Bazel](https://bazel.build/?utm_source=monorepo.tools), Microsoft has [Lage](https://microsoft.github.io/lage/?utm_source=monorepo.tools) and [Rush](https://rushstack.io/?utm_source=monorepo.tools), and others like [Lerna](https://lerna.js.org/?utm_source=monorepo.tools), [Nx](https://nx.dev/?utm_source=monorepo.tools) or [TurboRepo](https://turbo.build/repo) by Vercel.

**Advantages of using monorepo** as opposed to polyrepo(read more [here](https://monorepo.tools/) as I am merely summarising, but some parts of this link are not applicable anymore as TurboRepo has new updates. Ex: advisable to use pnpm, while npm and yarn are both available in TurboRepo, but the link stated TurboRepo only supports npm. Still, this link can give you a general overview of what a monorepo offers):

1\. **A single repository** containing multiple distinct projects with well-defined relationships(if there are no relationships between them, it‚Äôs not a monorepo but a polyrepo);

![Monorepo and Polyrepo](https://miro.medium.com/v2/resize:fit:1028/format:webp/1*vxjbrgoJOQWzh_JrSt4qEw.png)

**2\. Allow more team autonomy/mobility:** The industry has changed ‚Äî teams that build different functions/modules would want to make their own decisions about what libraries they‚Äôll use and who can contribute or use their code. They also want to verify if others‚Äô changes are safe.

**3\. Reduce code duplication:** if teams work in isolation, they will likely rewrite code‚Ä¶which is not optimal.

**4\. No more** `**create-xxx-app**` **within the code base**

**5\. One version of everything:** say there is a bug in a shared library; they need to apply the change to every repo, versioning issue, and update packages‚Ä¶monorepo controls all modules in one codespace.

**6\. Everything works together at every commit**
